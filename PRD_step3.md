# PRD: 냉장고 파먹기 웹 애플리케이션 - 3단계 (사용자 프로필 및 저장)

## 1. 개요
* **목표:** 사용자가 마음에 든 레시피를 자신의 계정에 저장하고, 언제든지 다시 꺼내볼 수 있도록 개인화된 경험을 제공합니다.
* **주요 기능:**
    * 사용자 회원가입 및 로그인 (인증 시스템)
    * 생성된 레시피 북마크(저장) 기능
    * '내 프로필 (My Page)'에서 저장된 레시피 목록 조회 및 관리 (삭제)

## 2. 사용자 시나리오
1. 아직 로그인하지 않은 사용자가 마음에 드는 레시피를 발견하고 하트 모양의 '저장' 버튼을 누릅니다.
2. 애플리케이션은 사용자에게 회원가입 또는 로그인을 요구하는 모달/페이지를 띄웁니다.
3. 사용자가 Email/Password 또는 OAuth(구글, 카카오 등)를 통해 로그인합니다.
4. 로그인 후, 해당 레시피가 사용자의 데이터베이스에 저장됩니다. (하트 아이콘 활성화)
5. 사용자는 상단 네비게이션 바에서 '내 프로필' 또는 '저장한 레시피' 메뉴로 이동합니다.
6. 지금까지 저장한 레시피들이 갤러리/리스트 형태로 모여 있습니다.
7. 사용자는 저장된 레시피를 클릭해 다시 상세 내용을 확인하거나, 더 이상 보지 않을 레시피는 목록에서 삭제할 수 있습니다.

## 3. 기능 요구사항
### 3.1. 프론트엔드 (UI/UX)
* **인증 UI:** 로그인, 회원가입, 비밀번호 찾기 폼.
* **저장 인터랙션:** 하트 아이콘 토글 애니메이션 등 직관적인 저장/취소 피드백.
* **마이페이지 뷰:** 저장된 레시피들을 카드 그리드나 리스트 뷰로 보여주는 화면 구현. 빈 화면(Empty State, "아직 저장한 레시피가 없습니다.") 처리.

### 3.2. 백엔드 / 데이터베이스 (기능 확장)
* **인증 시스템 (Authentication):** JWT, Session 기반 인증 또는 Firebase Auth, Supabase Auth 등 BaaS 솔루션 활용.
* **데이터베이스(DB) 설계:**
    * `User` 테이블: id, email, password_hash, created_at 등
    * `Recipe` 테이블: id, title, ingredients, instructions, created_at 등
    * `User_Recipe` 테이블 (다대다 관계 또는 Bookmark 테이블): user_id, recipe_id 매핑
* **CRUD API 구현:** 레시피 저장(Create), 저장된 목록 조회(Read), 저장 취소/삭제(Delete)를 수행하는 서버 API 엔드포인트 구축.

## 4. 비기능 요구사항
* **보안:** 사용자 비밀번호의 안전한 해싱(Bcrypt 등), 접근 권한(Authorization) 제어를 통한 개인 데이터 보호.
* **데이터 무결성:** 사용자가 계정을 탈퇴할 경우 해당 계정의 저장 데이터(북마크 내역 등)가 올바르게 처리(Soft Delete 혹은 Cascade)되도록 설계.
* **확장성:** 추후에 '사용자 간 레시피 공유'나 '별점 부여' 등의 기능이 추가될 수 있도록 유연한 DB 스키마 유지.

## 5. 성공 지표 (Metrics)
* 회원 가입 전환율 (레시피를 보고 로그인을 시도하는 비율).
* 사용자 당 평균 레시피 저장 횟수 (기능 활성화 지표).
* 재방문율 (Day-1, Day-7 Retention) - 저장된 레시피를 보기 위해 사용자가 다시 접속하는지 측정.
